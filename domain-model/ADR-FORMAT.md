# ADR 格式

ADRs 位于 `docs/adr/` 中, 并使用顺序编号: `0001-slug.md`, `0002-slug.md` 等.

懒惰创建 `docs/adr/` 目录 -- 仅在需要第一个 ADR 时.

## 模板

```md
# {决策的简短标题}

{1-3 句话: 上下文是什么, 我们决定了什么, 以及为什么.}
```

就是这样. ADR 可以是一个段落. 价值在于记录*做出了*决策以及*为什么* -- 而不是填写部分.

## 可选部分

仅在它们增加真正价值时包含这些. 大多数 ADRs 不需要它们.

- **Status** frontmatter(`proposed | accepted | deprecated | superseded by ADR-NNNN`) -- 在重新审视决策时有用
- **Considered Options** -- 仅当被拒绝的替代方案值得记住时
- **Consequences** -- 仅当需要指出非显而易见的下游效应时

## 编号

扫描 `docs/adr/` 以查找最高的现有编号并加一.

## 何时提供 ADR

所有这三个都必须为真:

1.** 难以逆转** -- 以后改变主意的成本是有意义的
2.** 没有上下文令人惊讶** -- 未来的读者会查看代码并想知道 "他们为什么要这样做?"
3.** 真实权衡的结果** -- 有真正的替代方案, 你出于特定原因选择了一个

如果决策容易逆转, 跳过它 -- 你只会逆转它. 如果它不令人惊讶, 没有人会想知道为什么. 如果没有真正的替代方案, 除了 "我们做了显而易见的事情" 之外没有什么可记录的.

### 什么符合条件

- **架构形状.* * "我们使用 monorepo." "写入模型是事件源的, 读取模型投影到 Postgres."
- **上下文之间的集成模式.* * "Ordering 和 Billing 通过领域事件通信, 而不是同步 HTTP."
- **带有锁定的技术选择.* * 数据库, 消息总线, 身份验证提供程序, 部署目标. 不是每个库 -- 只是那些需要一个季度才能交换的库.
- **边界和范围决策.* * "Customer 数据由 Customer 上下文拥有; 其他上下文仅通过 ID 引用它." 明确的否定与肯定一样有价值.
- **有意偏离明显路径.* * "我们使用手动 SQL 而不是 ORM, 因为 X." 任何合理的读者会假设相反的东西. 这些阻止下一个工程师 "修复" 有意为之的东西.
- **代码中不可见的约束.* * "由于合规要求, 我们不能使用 AWS." "由于合作伙伴 API 合同, 响应时间必须低于 200ms."
- **拒绝的替代方案, 当拒绝不明显时.* * 如果你考虑了 GraphQL 并出于微妙原因选择了 REST, 记录它 -- 否则六个月后有人会再次建议 GraphQL.
