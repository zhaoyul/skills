# 范围外知识库

仓库中的 `.out-of-scope/` 目录存储被拒绝的功能请求的持久记录. 它有两个目的:

1.** 机构记忆** — 为什么功能被拒绝, 这样当 issue 关闭时推理不会丢失
2.** 去重** — 当新 issue 与先前的拒绝匹配时, 技能可以浮出先前的决策, 而不是重新争论

## 目录结构

```
.out-of-scope/
├── dark-mode.md
├── plugin-system.md
└── graphql-api.md
```

每个**概念**一个文件, 而不是每个 issue. 请求相同内容的多个 issues 分组在一个文件下.

## 文件格式

文件应该以轻松, 可读的风格编写 -- 更像是一个简短的设计文档而不是数据库条目. 使用段落, 代码示例和例子使推理清晰且对第一次遇到它的人有用.

```markdown
# 深色模式

此项目不支持暗模式或面向用户的主题.

## 为什么这是范围外的

渲染管道假设在 `ThemeConfig` 中定义单个调色板. 支持多个主题需要:

- 包装整个组件树的主题上下文提供者
- 每个组件的主题感知样式解析
- 用户主题偏好的持久层

这是一个重大的架构更改, 不符合项目对内容创作的关注.
主题是嵌入或重新分发输出的下游消费者的关注点.

```ts
// 当前的 ThemeConfig 接口不是为运行时切换设计的:
interface ThemeConfig {
  colors: ColorPalette; // 单个调色板, 在构建时解析
  fonts: FontStack;
}
```

## 先前的请求

- #42 — "添加深色模式支持"
- #87 — "为无障碍提供夜间主题"
- #134 — "深色主题选项"
```

### 命名文件

为概念使用简短的, 描述性的 kebab-case 名称: `dark-mode.md`, `plugin-system.md`, `graphql-api.md`. 名称应该足够可识别, 以便浏览目录的人无需打开文件就能理解被拒绝的内容.

### 编写原因

原因应该是实质性的 -- 不是 "我们不想要这个", 而是为什么. 好的原因引用:

- 项目范围或哲学("此项目专注于 X; 主题是下游关注点")
- 技术约束("支持这个需要 Y, 这与我们的 Z 架构冲突")
- 战略决策("我们选择使用 A 而不是 B, 因为...")

原因应该是持久的. 避免引用临时情况("我们现在太忙了") -- 那些不是真正的拒绝, 它们是延期.

## 何时检查 `.out-of-scope/`

在 triage 期间(步骤 1: 收集上下文), 阅读 `.out-of-scope/` 中的所有文件. 评估新 issue 时:

- 检查请求是否匹配现有的范围外概念
- 匹配是通过概念相似性, 而不是关键字 -- "night theme" 匹配 `dark-mode.md`
- 如果有匹配, 向维护者浮出它: "这类似于 `.out-of-scope/dark-mode.md` — 我们之前拒绝了这个, 因为 [原因]. 你仍然有同样的感觉吗?"

维护者可能:

- **确认** — 新 issue 被添加到现有文件的 "Prior requests" 列表, 然后关闭
- **重新考虑** — 范围外文件被删除或更新, issue 通过正常 triage 继续
- **不同意** — issues 相关但不同, 通过正常 triage 继续

## 何时写入 `.out-of-scope/`

仅当**enhancement**(不是 bug)被拒绝为 `wontfix` 时. 流程:

1. 维护者决定功能请求超出范围
2. 检查是否已存在匹配的 `.out-of-scope/` 文件
3. 如果是: 将新 issue 附加到 "Prior requests" 列表
4. 如果否: 创建一个新文件, 包含概念名称, 决策, 原因和第一个先前请求
5. 在 issue 上发布评论解释决策并提到 `.out-of-scope/` 文件
6. 用 `wontfix` 标签关闭 issue

## 更新或删除范围外文件

如果维护者改变了对先前被拒绝概念的想法:

- 删除 `.out-of-scope/` 文件
- 技能不需要重新打开旧 issues -- 它们是历史记录
- 触发重新考虑的新 issue 通过正常 triage 继续
