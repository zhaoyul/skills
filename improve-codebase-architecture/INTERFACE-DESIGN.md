# 接口设计

当用户想要为选定的深化候选探索替代接口时，使用此并行子代理模式。基于 "Design It Twice"（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**module**、**interface**、**seam**、**adapter**、**leverage**。

## 流程

### 1. 界定问题空间

在生成子代理之前，为选定的候选编写一个面向用户的问题空间说明：

- 任何新接口需要满足的约束
- 它将依赖的依赖项，以及它们属于哪个类别（见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的说明性代码草图，以使约束具体化——不是提案，只是使约束具体的一种方式

向用户展示这一点，然后立即进入第 2 步。用户在子代理并行工作时阅读和思考。

### 2. 生成子代理

使用 Agent 工具并行生成 3+ 个子代理。每个必须为深化模块生成一个**根本不同**的接口。

用单独的技术简报（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、seam 后面是什么）提示每个子代理。简报独立于第 1 步中面向用户的问题空间说明。给每个代理一个不同的设计约束：

- 代理 1："最小化接口——目标是最多 1-3 个入口点。最大化每个入口点的杠杆。"
- 代理 2："最大化灵活性——支持许多用例和扩展。"
- 代理 3："优化最常见的调用者——使默认情况变得微不足道。"
- 代理 4（如果适用）："围绕端口和适配器设计跨 seam 依赖项。"

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇和 CONTEXT.md 词汇，以便每个子代理以与架构语言和项目领域语言一致的方式命名事物。

每个子代理输出：

1. 接口（类型、方法、参数——加上不变量、顺序、错误模式）
2. 显示调用者如何使用它的使用示例
3. 实现在 seam 后面隐藏了什么
4. 依赖策略和适配器（见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——杠杆在哪里高，在哪里薄

### 3. 呈现和比较

依次呈现设计，以便用户可以吸收每一个，然后用散文比较它们。通过**深度**（接口处的杠杆）、**局部性**（变更集中在哪里）和**seam 放置**进行对比。

比较后，给出你自己的推荐：你认为哪个设计最强以及为什么。如果不同设计的元素可以很好地结合，提出一个混合方案。要有主见——用户想要一个强有力的解读，而不是菜单。
