# 深化

如何在给定其依赖项的情况下安全地深化浅层模块集群. 假设 [LANGUAGE.md](LANGUAGE.md) 中的词汇 -- **module**,** interface**,** seam**,** adapter**.

## 依赖类别

在评估深化候选时, 对其依赖项进行分类. 类别决定了如何跨其 seam 测试深化模块.

### 1. 进程内

纯计算, 内存状态, 无 I/O. 总是可深化 -- 合并模块并直接通过新接口进行测试. 不需要适配器.

### 2. 本地可替代

具有本地测试替身的依赖项(Postgres 的 PGLite, 内存文件系统). 如果替身存在则可深化. 深化模块在测试套件中运行替身进行测试. seam 是内部的; 模块的外部接口没有端口.

### 3. 远程但拥有(端口和适配器)

跨网络边界的你自己的服务(微服务, 内部 API). 在 seam 处定义一个**端口**(接口). 深度模块拥有逻辑; 传输作为**适配器**注入. 测试使用内存适配器. 生产使用 HTTP/gRPC/队列适配器.

推荐形状:_ "在 seam 处定义一个端口, 为生产实现一个 HTTP 适配器, 为测试实现一个内存适配器, 以便逻辑位于一个深度模块中, 即使它跨网络部署."_

### 4. 真正的外部(模拟)

你不控制的第三方服务(Stripe, Twilio 等). 深化模块将外部依赖作为注入的端口; 测试提供模拟适配器.

## Seam 纪律

- **一个 adapter 意味着假设的 seam. 两个 adapters 意味着真实的 seam.* * 除非至少两个适配器是合理的(通常是生产 + 测试), 否则不要引入端口. 单适配器 seam 只是间接.
- **内部 seams vs 外部 seams.* * 深度模块可以有内部 seams(对其实现是私有的, 由其自己的测试使用)以及其接口处的外部 seam. 不要仅仅因为测试使用它们就通过接口暴露内部 seams.

## 测试策略: 替换, 而不是分层

- 一旦深化模块接口处的测试存在, 浅层模块上的旧单元测试就变成浪费 -- 删除它们.
- 在深化模块的接口处编写新测试.** 接口是测试表面**.
- 测试断言通过接口的可观察结果, 而不是内部状态.
- 测试应该在内部重构中幸存 -- 它们描述行为, 而不是实现. 如果测试必须在实现更改时更改, 它就是在越过接口进行测试.
